# ADR-0002: Search and fetch contract

## Status

Accepted

## Context

Research clients need a small, predictable, citation-friendly contract.

A broad GitHub API surface is not desirable for Deep Research-style usage because it mixes repository reading with operational actions and exposes too many implementation details.

The research interface should behave like a document retrieval system: `search` discovers relevant documents and `fetch` retrieves one document by an identifier returned by `search`.

## Decision

The public MCP contract starts with exactly two research tools:

- `search(query, repository, limit)`
- `fetch(id)`

`search` must return compact results that are safe to render, cite, and pass back to `fetch`.

`fetch` must only accept IDs generated by this service format and must re-check the repository allowlist before returning content.

## Search result fields

Each search result should include:

- `id`: stable document identifier.
- `title`: readable title, usually the file path or heading.
- `url`: canonical GitHub URL.
- `text`: short relevant snippet or summary.
- `metadata.repository`: repository full name.
- `metadata.path`: repository-relative path.
- `metadata.ref`: branch, tag, or commit ref.
- `metadata.sha`: optional blob sha.
- `metadata.source`: source system, initially `github`.

## Fetch result fields

A fetched document should include:

- `id`: same stable document identifier.
- `title`: readable title.
- `url`: canonical GitHub URL.
- `text`: full bounded document text.
- `metadata`: repository, path, ref, sha, and source.

## Identifier rules

Document IDs must be deterministic, parseable by the service, stable enough for search/fetch round-trips, and treated as untrusted user input when received by `fetch`.

`fetch` must validate the parsed repository against the allowlist every time.

## Bounds

Tool output must be bounded.

Search should return a limited number of compact results.
Fetch should cap document size and either truncate safely or return an explicit error telling the client how to narrow the request.

## Consequences

The service favors a small document retrieval contract over a broad repository automation API.

This keeps the integration safer and makes it easier to connect through `central-mcp-gateway` as a research-only upstream.
